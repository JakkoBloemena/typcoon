# Retro — tick 2026-07-22 #1 (adoption board, first tick)

Written for a reader in another company. Three lessons, one pattern confirmation.

## 1. A live Shareholder collapses decision latency — use it

This tick ran inside the Shareholder's own /ceo channel session. The payments
decision package (002) was written by a CEO lane and presented to the Shareholder
*in the same tick*; the decision (deferral, price, paywall behavior) was recorded as
an ADR minutes later. The same package parked as `blocked — awaiting Shareholder`
would normally wait days. **Lesson:** when a tick runs with the Shareholder present,
the dispatcher should present ready decision packages before closing the tick,
instead of leaving them in `blocked`. The framework's tick skill could name this
explicitly.

## 2. Feature work invalidates copy claims — reconcile in the same assignment

A chain of three assignments (001 → 008 → 011) each fixed privacy copy that the
previous change had made stale: 001 fixed claims broken by the accounts feature;
006 (analytics) immediately broke the "geen tracking" and "geen gegevens naar een
server" claims 001 had just classified as honest; 008 fixed the first, and its lane
surfaced the second, which became 011. Every fix was found by a lane's honest
repo-wide search — the system worked, but serially, three dispatches deep.
**Lesson:** any assignment that changes *what the product does* (data flows
especially) should carry an acceptance criterion "reconcile every user-facing claim
this change touches" — cheaper than a follow-up assignment per stale claim.

## 3. Record overridden recommendations inside the ADR

The Shareholder kept the silent free unlock against the CEO's recommendation
(pre-order intent CTA). The ADR (decisions/002 §4) records both the choice and the
CEO's reservation. **Lesson:** when a recommendation is overridden, the ADR must
carry the dissent — a future agent reading only the decision would otherwise
re-propose the rejected option as if new, or silently "fix" the chosen one.

## Pattern confirmation

Worktree-per-lane parallelism (5 concurrent lanes at peak) produced zero file
collisions and clean merges; the one predicted collision (006 vs 001 on index.html)
was correctly serialized by the priority rule and dispatched mid-tick after 001
merged. The mid-tick claim commit (006, then 007/008, then 011) kept the ledger
honest at every step.
